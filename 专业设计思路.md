## 第一部分：数据库设计

### 一、现有表结构修改

---

#### 1. userAccount（用户账号表）

> **无新增字段，与原表完全一致。**

```sql
CREATE TABLE `userAccount` (
    `id` bigint NOT NULL AUTO_INCREMENT COMMENT '用户ID',
    `username` varchar(50) NOT NULL COMMENT '登录账号',
    `password` varchar(100) NOT NULL COMMENT 'BCrypt加密密码',
    `nickname` varchar(50) NOT NULL COMMENT '昵称',
    `email` varchar(100) DEFAULT NULL COMMENT '邮箱',
    `emailVerified` tinyint DEFAULT 0 COMMENT '邮箱认证 0-未认证 1-已认证',
    `oauthType` varchar(20) DEFAULT NULL COMMENT '第三方登录类型(gitee/github等)',
    `oauthOpenid` varchar(100) DEFAULT NULL COMMENT '第三方openid',
    `role` varchar(10) NOT NULL DEFAULT 'user' COMMENT '用户角色：user-普通用户 admin-管理员',
    `status` tinyint DEFAULT 1 COMMENT '账号状态 1-正常 0-禁用',
    `answerNum` int DEFAULT 0 COMMENT '总做题数',
    `correctNum` int DEFAULT 0 COMMENT '总做对题数',
    `createTime` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP,
    `updateTime` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    `isDeleted` tinyint DEFAULT 0,
    PRIMARY KEY (`id`),
    UNIQUE KEY `ukUsername` (`username`),
    UNIQUE KEY `ukEmail` (`email`),
    UNIQUE KEY `ukOauth` (`oauthType`,`oauthOpenid`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用户账号表';
```

#### 2. question（试题主表）

> **新增 4 个组卷专用字段。**
>
> 注：`subjectId` 和 `questionTypeId` 已移除 —— 原表已有 `subject` varchar 和 `type` tinyint 字段，无需冗余。学科和题型通过新增的 `subject` / `questionType` 字典表 JOIN 查询即可。

| 新增字段 | 类型 | 说明 |
|---|---|---|
| `discrimination` | tinyint DEFAULT 0 | 区分度 1-5 |
| `calcLevel` | tinyint DEFAULT 0 | 计算量等级 1-3 |
| `examFrequency` | int DEFAULT 0 | 考频 0-100 |
| `gradeStageId` | bigint DEFAULT NULL | 学段ID |

```sql
CREATE TABLE `question` (
    `id` bigint NOT NULL AUTO_INCREMENT COMMENT '主键ID',
    `questionMd5` char(32) NOT NULL COMMENT '题干MD5指纹，用于查重',
    `type` tinyint NOT NULL COMMENT '题型 1-单选 2-多选 3-填空 4-简答',
    `subject` varchar(50) NULL COMMENT '学科',
    `chapter` varchar(100) DEFAULT NULL COMMENT '章节',
    `difficulty` tinyint NULL COMMENT '难度 1-易 2-中 3-难',
    `knowledgePoints` varchar(500) DEFAULT NULL COMMENT '知识点，逗号分隔',
    `tags` json DEFAULT NULL COMMENT '题目标签（字符串数组）',
    `content` text NOT NULL COMMENT '题干内容',
    `options` json DEFAULT NULL COMMENT '选项JSON',
    `answer` text NOT NULL COMMENT '标准答案',
    `analysis` text DEFAULT NULL COMMENT '解析',
    `creatorId` bigint NOT NULL COMMENT '创建人ID',
    `status` tinyint DEFAULT 1 COMMENT '状态 1-草稿 2-已发布 3-停用',
    `correctCount` bigint DEFAULT 0 COMMENT '做对次数',
    `totalCount` bigint DEFAULT 0 COMMENT '做题总次数',
    `accuracy` decimal(5,2) DEFAULT 0.00 COMMENT '正确率',
    `createTime` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP,
    `updateTime` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    -- 新增：组卷模块专用字段
    `discrimination` tinyint DEFAULT 0 COMMENT '区分度 1-5',
    `calcLevel` tinyint DEFAULT 0 COMMENT '计算量等级 1-3',
    `examFrequency` int DEFAULT 0 COMMENT '考频 0-100',
    `gradeStageId` bigint DEFAULT NULL COMMENT '学段ID',
    `isDeleted` tinyint DEFAULT 0,
    PRIMARY KEY (`id`),
    UNIQUE KEY `ukQuestionMd5` (`questionMd5`),
    KEY `idxCreator` (`creatorId`),
    KEY `idxTypeDifficulty` (`type`,`difficulty`),
    KEY `idxSubjectChapter` (`subject`,`chapter`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='试题主表';
```

### 二、新增表结构

---

#### 1. paperStrategy（组卷策略表）

> 核心表：存储自动/AI组卷的规则模板。

| 字段 | 类型 | 说明 |
|---|---|---|
| `questionTypeConfig` | json DEFAULT NULL | 各题型数量/分值配置，如 `[{"type":1,"count":10,"score":5},{"type":2,"count":5,"score":10}]` |
| `difficultyConfig` | json DEFAULT NULL | 各难度占比配置，如 `[{"level":1,"ratio":0.2},{"level":2,"ratio":0.5},{"level":3,"ratio":0.3}]` |
| `knowledgePointScope` | json DEFAULT NULL | 知识点范围，如 `["二次函数","一元一次方程"]` |

```sql
CREATE TABLE `paperStrategy` (
    `id` bigint NOT NULL AUTO_INCREMENT COMMENT '策略ID',
    `strategyName` varchar(100) NOT NULL COMMENT '策略名称',
    `userId` bigint NOT NULL COMMENT '创建人ID',
    `totalScore` int NOT NULL DEFAULT 100 COMMENT '目标总分',
    `difficultyAvg` tinyint DEFAULT 3 COMMENT '平均难度 1-5',
    `duration` int DEFAULT NULL COMMENT '答题时长（分钟）',
    `questionTypeConfig` json DEFAULT NULL COMMENT '各题型数量/分值配置',
    `difficultyConfig` json DEFAULT NULL COMMENT '各难度占比配置',
    `knowledgePointScope` json DEFAULT NULL COMMENT '知识点范围',
    `isDefault` tinyint DEFAULT 0 COMMENT '是否默认策略 0-否 1-是',
    `createTime` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP,
    `updateTime` datetime NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    `isDeleted` tinyint DEFAULT 0,
    PRIMARY KEY (`id`),
    KEY `idxUserId` (`userId`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='组卷策略表';
```

#### 1. errorBook（错题本主表）

> 记录用户每道错题的核心信息，支持按错误类型、知识点、复习状态筛选排序。

```sql
CREATE TABLE `errorBook` (
```

#### （一）基本组卷策略

**1. 组卷量化指标体系设计**

##### 量化指标定义

| 指标名称 | 取值范围 | 最优取值 | 最差取值 | 核心特殊约束（含硬约束/学段/学科适配） |
|---|---|---|---|---|
| **难度** | 1-5 级（整数） | 3 级（中等难度），适配 90% 以上试卷类型 | 1 级（过易）/5 级（过难），单一极值占比超 60% 则试卷无效 | 1. 仅在同学段内做 1-5 级相对量化；2. 整卷难度建议正态分布，避免单一级别扎堆；3. 练习卷侧重 1-3 级，正式选拔卷侧重 3-5 级 |

| **考频** | 0%-100%（保留 2 位小数，历史试卷出现频率） | 10%-30%，兼顾验证性与多样性 | ≥60%（高频扎堆）/0%（全新未使用题，无数据参考） | 1. 按「学段+学科」维度统计，不跨维度累计；2. 正式试卷单题考频≤20%，整卷高频题（≥40%）占比≤10%；3. 新授课练习卷可允许 0% 考频新题，占比≤50% |
| **考点关联数** | 正整数（≥1，关联知识点数量） | 2-3 个（1 核心考点 + 1-2 关联考点），兼顾广度与深度 | ≥5 个，考点过多导致题目偏杂、偏离考查目标 | 1. 基础题（难度 1-2）考点关联数≤2 个，突出核心；2. 压轴题（难度 4-5）可放宽至 3-4 个，考查综合应用；3. 所有关联考点必须属于当前学段教学大纲范围 |

##### 组卷量化指标权重设计基本框架和核心规则

本框架与规则基于学段固定、无需跨学科前置约束设计，适配 AI 自由调整权重的核心需求，无预设权重模板、无固定指标占比，仅划定权重设计的底层逻辑、结构边界和操作准则，既保证 AI 权重调配的灵活性，又锚定权重设计不偏离量化指标体系的科学逻辑，所有权重设计均围绕难度、正确率、区分度、计算量、考频、考点关联数 6 个量化指标展开（适配学段为一级硬约束，不参与权重分配，直接做题目筛选剔除）。

**一、权重设计的基本框架**

采用三层递进式架构，从前置约束到核心分配对象，再到灵活调配维度，层层界定权重设计的范围和落地方式，支撑人工自定义与 AI 自由调权重的双重需求，框架无任何数值化权重预设，仅明确各层的功能和设计对象。

- **第一层：硬约束前置层（权重设计的前提，无调配空间）**
  - 设计前提：所有权重分配与调配，均以学段 100% 匹配、当前学科标签锁定为基础，未满足该前提的题目直接剔除，不参与权重筛选；
  - 指标底限：权重设计不得突破 6 个量化指标的最低合规硬约束（如正式卷区分度≥0.3、小 1-3 年级无 3 级计算量），AI 调配权重时，系统自动校验指标硬约束，若权重调配指向违规指标区间，直接驳回并提示。

- **第二层：核心指标权重池（权重设计的唯一分配对象，总分锚定）**
  - 权重池构成：将难度、正确率、区分度、计算量、考频、考点关联数 6 个量化指标作为权重设计的唯一对象，形成独立权重池，无其他额外指标参与分配；
  - 总分锚定：权重池内所有指标的权重值之和固定为 100%，为权重设计的核心锚点，人工/AI 调配均需满足该锚点，系统做实时校验，不满足则无法保存/执行权重方案。

- **第三层：多维调配层（权重设计的落地维度，AI 自由调配的核心载体）**
  - 整体维度调配：对 6 个指标进行整卷级的权重分配，即给每个指标分配一个全局权重值，适用于整卷有统一考查要求的组卷需求；
  - 细分维度调配：对单个量化指标做学科子维度拆解后再分配权重，适用于精细化、个性化组卷需求（AI 组卷的核心调配方式），例如：
    - 难度指标：按知识点/题型拆解（如"函数知识点难度"分配权重、"选择题难度"分配权重）；
    - 正确率指标：按学生群体/知识点拆解（如"本班薄弱知识点正确率"分配权重、"培优学生正确率"分配权重）；
  - 动态维度调配：权重值可根据组卷过程中的题目匹配情况动态调整，适用于题库题目数量不足的场景，AI 可按指标优先级自动降低低优先级指标的权重，实现指标降级匹配。

**二、权重设计的核心基本规则**

规则分为基础合规规则、权重调配规则、指标联动规则、落地执行规则四大类，覆盖权重设计的底线要求、操作方式、与量化指标的衔接、实际组卷的落地逻辑，所有规则为强制性要求，人工/AI 设计权重均需遵循，无例外情况。

**（一）基础合规规则：权重设计的"及格线"，不可突破**

| 规则 | 说明 |
|---|---|
| 权重值范围规则 | 单个指标的权重值取值范围为 0%-100%，支持整数/小数（保留 1-2 位小数，精度可系统配置），不允许出现负值或超出 100% 的数值 |
| 总分闭环规则 | 6 个量化指标的权重值之和必须严格等于 100%，系统实时计算权重总和，不足/超出时给出明确提示，且无法保存权重方案、无法触发组卷算法 |
| 硬约束排他规则 | 权重设计仅针对 6 个量化指标的合规取值区间，若某指标存在硬约束（如计算量 3 级占比≤30%），AI 不得为该指标的违规取值区间分配任何权重，系统自动屏蔽违规区间的权重调配可能 |
| 指标有效性规则 | 权重值为 0% 的指标，代表该指标不参与本次组卷的题目筛选，系统将直接忽略该指标的量化规则（但仍需满足硬约束），AI 可根据需求将任意非核心指标的权重调至 0% |

**（二）权重调配规则：权重设计的"灵活性准则"，支撑自由调配**

| 规则 | 说明 |
|---|---|
| 无级调配规则 | 对 6 个指标的权重分配支持 0%-100% 的无级调整，无阶梯式比例限制（如可给难度分配 37.5%、区分度分配 22.3%），AI 可根据组卷需求的量化描述，精准分配任意权重值，无需拘泥于固定比例 |
| 细分拆解规则 | 单个量化指标可按学科/教学/学生的子维度无限拆解，拆解后的子维度权重值之和等于该指标的全局权重值 |
| 优先级关联规则 | 权重值与指标筛选优先级正相关，即指标的权重值越高，在组卷算法中的筛选优先级越高 |
| 动态调整规则 | 当题库中符合当前权重方案的题目数量不足时，AI 可按指标权重从低到高依次降低指标权重（或调至 0%），实现指标降级匹配，且每一次动态调整均需满足"总分 100%"规则，同时向用户明确提示调整内容 |

**（三）指标联动规则：权重设计与量化指标体系的"衔接准则"，避免偏离**

| 规则 | 说明 |
|---|---|
| 强联动指标权重协同规则 | 对于存在强联动关系的指标（难度&正确率、难度&考点关联数、难度&区分度），AI 调配权重时，需保证联动指标的权重协同适配 |
| 指标约束与权重匹配规则 | 若某指标存在场景化约束（如正式卷区分度≥0.3、练习卷难度以 1-3 级为主），AI 需为该指标的合规约束区间分配更高权重，为非合规区间分配 0 权重 |
| 无联动独立调配规则 | 无强联动关系的指标（如考频&计算量），可独立调配权重 |

**（四）落地执行规则：权重设计与组卷算法/系统功能的"适配准则"，保证可落地**

| 规则 | 说明 |
|---|---|
| 算法适配规则 | 贪心算法：按指标权重从高到低依次筛选题目；遗传算法：将指标权重作为适应度函数的系数 |
| 权限管控规则 | 教师：可自定义设计权重方案、保存为个人模板、调用个人模板；管理员：可查看所有用户的权重方案、做批量管理，可设置系统级权重基础规则 |
| 模板化保存规则 | 支持将自定义/AI 生成的权重方案保存为模板，模板仅记录指标权重分配方式/数值，无预设模板，所有模板均为用户自定义生成 |
| 缓存适配规则 | AI 生成/人工设计的高频权重方案（近 7 天使用≥3 次），自动缓存至 Redis，结合 Caffeine 本地缓存 |

**三、补充：AI 自由调权重的专属设计原则**

| 原则 | 说明 |
|---|---|
| 需求量化映射原则 | AI 需将用户的自然语言组卷需求精准转化为量化指标的权重分配依据，需求的核心诉求对应更高权重，次要诉求对应低权重，无诉求则调至 0% |
| 个性化数据联动原则 | AI 调配权重时，需结合用户的个性化数据（错题本、学习画像、班级成绩数据），将个性化数据中的核心特征（如薄弱知识点、错题难度）对应指标分配更高权重 |
| 题库数据适配原则 | AI 在调配权重前，需先检索题库中各指标的题目数量，若某指标的合规题目数量极少，提前降低该指标权重，避免后续动态调整的频次过高 |

**2. 高效组卷算法的设计与实现**

采用贪心算法实现基础组卷、遗传算法实现高精度组卷，组卷成功率≥95%。

**一、方案拆解**

| 场景 | 核心算法 | 说明 |
|---|---|---|
| **手动组卷** | 纯贪心 + 轻量遗传微调 | 纯贪心为核心保证毫秒级响应；轻量遗传微调仅在用户确认"基础卷"后做 1-2 轮极简迭代（替换 1-2 道题），后台异步完成 |
| **AI 组卷** | 贪心初始化 + 遗传迭代 | 贪心快速生成 50-80 组"基础合格卷"作为初始种群（而非随机生成），遗传迭代逼近全局最优解 |

**二、落地时的 2 个关键注意点**

1. 手动组卷的遗传微调要"轻量"：仅针对"已选的基础卷"做局部调整，不重新遍历全题库，迭代次数控制在 5-10 次，变异率设为 0.01-0.05，确保微调耗时＜1 秒
2. AI 组卷的贪心初始化要保留"多样性"：生成初始种群时，用不同的贪心优先级（如第 1 组先优先难度、第 2 组先优先知识点、第 3 组先优先分值），避免所有初始卷都高度相似

**3. 性能指标要求**

组卷功能支持万级别题库的题目处理量，单用户组卷响应时间≤100ms；支持≥10 用户同时组卷，高并发场景下无数据错乱；当符合条件题目数量不足时，算法支持按指标优先级降级匹配，并向用户给出明确的操作提示。

**一、核心优化：数据层（占提速效果的 70%）**

1. **题库预分区 + 结构化存储**
   - 预分区逻辑：学段 → 学科 → 知识点 → 题型 → 难度
   - MySQL 复合索引 + Redis 缓存高频迷你题池

2. **数据预处理 + 冗余存储**
   - 提前计算每题关键指标并冗余存储
   - 提前标记作废题、重复题并剔除

3. **缓存策略**

| 缓存对象 | 缓存介质 | 过期策略 | 提速效果 |
|---|---|---|---|
| 迷你题池 | Redis | 7 天过期（题池更新时主动刷新） | 数据库查（50ms）→ 内存查（1ms） |
| 组卷条件+结果 | Redis | 1 小时过期（相同条件直接返回结果） | 重复组卷请求无需重新计算 |
| 算法中间结果 | 本地内存（JVM） | 请求结束后释放 | 避免同一次组卷内重复筛选题目 |

**二、算法层优化**

| 算法 | 优化策略 |
|---|---|
| **贪心算法（手动组卷）** | 极致剪枝 + 一步到位：提前剪枝、单轮遍历（多轮合并为一次）、放弃遗传微调（优先保证响应速度≤100ms） |
| **遗传算法（AI 组卷）** | 极致缩小解空间 + 早停：先缩小到 200 题以内、轻量化参数（种群 30、迭代 20、变异率 0.01）、早停机制 |

**三、工程层优化**

| 优化项 | 措施 |
|---|---|
| 并发控制 | 无状态服务设计、独立线程池（核心 20、最大 50）、MySQL 连接池 20、Redis 连接池 50、读操作无锁化 |
| 代码细节 | HashSet/HashMap 做条件判断、Protocol Buffers 序列化、非核心逻辑异步化 |

**四、降级匹配策略**

- 提前预判：组卷前统计迷你题池数量，不足则直接触发降级逻辑
- 优先级固化：题型数量 > 总分值 > 难度范围 > 知识点分布
- 明确提示：返回结构化提示，清晰告知用户调整内容

#### （二）用户手动组卷

1. 提供可视化手动组卷表单，支持试卷方向、试卷类型、试卷总分、答题时长、整体难度的模糊选择；支持知识点精准筛选、各题型数量或分值自定义、不同难度题占比精细化配置

#### （三）AI 辅助组卷

1. 提供 AI 组卷智能对话界面，用户可通过自然语言描述组卷需求；系统将标准化处理后的用户个人学习信息、错题本错题等个性化信息，结合不同学科、学段、试卷类型的专属标准化模板，预填为 AI 对话提示词，同时在提示词中明确标准组卷要求格式

2. AI 大模型基于用户描述与预填提示词进行分析，输出符合标准格式的组卷要求；系统将 AI 生成的组卷要求回显给用户，支持用户对规则进行二次编辑与合规性校验，用户可选择确认或重新生成组卷要求

**确保 AI 返回标准化格式的措施：**

**一、源头约束：提示词层面"强制标准化"**
- 格式要求"极致明确"：字段清单、字段类型、禁止内容、加示例
- 增加"格式校验规则"的提示，强化 AI 对格式的重视

**二、过程约束：API 参数层面"硬限制"**
- 强制指定输出格式：设置 `response_format={"type":"json_object"}`（OpenAI）/ `result_format="json"`（通义千问/文心一言）
- 降低随机性：`temperature` 设为 0.1~0.3
- 限制输出长度：`max_tokens` 设为刚好能容纳的长度

**三、结果兜底：系统层面"校验+清洗"**
- 第一步：纯文本清洗（正则提取 JSON）
- 第二步：格式+字段校验（确保可解析）
- 第三步：容错处理（自动重试 2-3 次，失败则切换手动编辑）

3. 若用户确认组卷方案，系统将 AI 生成的标准化组卷要求交由组卷算法处理；若 AI 生成失败、内容异常或接口调用超时，系统自动触发重试机制（最多 3 次），重试失败则切换为手动组卷模式并给出明确提示

4. 保留用户与 AI 的组卷对话记录及 AI 生成的组卷策略，支持历史记录的查询与组卷策略的一键复用，同时实现个性化信息的动态更新，错题本、个人学习画像更新后，提示词中的预填信息实时同步

#### （四）试卷导出

2. 实现导出预览与批量导出功能，用户导出前可在线预览试卷排版效果，支持对预览结果进行简单调整；同时支持多份试卷的批量导出

2. 基于 ECharts 实现统计数据的可视化展示，至少支持柱状图、饼图、折线图三种图表类型，图表具备维度筛选、钻取查看详情的交互能力，支持图表的放大、下载；同时支持单用户、整体在该试卷对应知识点的多次考试成绩趋势分析

#### （六）模块通用技术要求

1. **权限控制**：基于 Sa-Token 实现精细化权限控制，教师仅可对本人创建的试卷进行操作，可将已发布试卷分享给指定班级/学生；管理员拥有全量操作权限

2. **缓存与异步处理**：利用 Redis 缓存高频组卷策略、试卷基础信息、高频题目数据，缓存有效期 24 小时；结合 Caffeine 本地缓存提升速度；大篇幅试卷（≥50 题）组卷、批量导出等耗时操作采用 RabbitMQ 实现异步处理，前端实时显示处理进度，完成后通过系统通知推送结果

3. **数据一致性与生命周期保障**：
   - 试卷状态切换规则：草稿 → 已发布 → 已归档 → 已停用
   - 已发布试卷不可直接编辑，仅支持复制后修改
   - 题目逻辑删除后，关联试卷对应题目标记为"题目已失效"
   - 教师试卷数据留存 90 天，学生考试记录留存 180 天，逾期自动清理

4. **跨设备同步与兼容性**：试卷配置、组卷策略、导出记录等数据支持跨设备同步；前端响应式适配 PC 端、平板端、手机端；导出格式兼容主流软件版本
